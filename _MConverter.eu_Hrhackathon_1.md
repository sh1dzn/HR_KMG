# 3. Техническое задание - AI-модуль {#техническое-задание---ai-модуль .unnumbered}

## 3.1 Компонент A: AI-оценка качества и релевантности целей {#компонент-a-ai-оценка-качества-и-релевантности-целей .unnumbered}

**3.1.1 Методология SMART**

|                |                           |                                                                                                         |
|----------------|---------------------------|---------------------------------------------------------------------------------------------------------|
| **Критерий**   | **Расшифровка**           | **Что проверяет AI**                                                                                    |
| S - Specific   | Конкретность              | Цель содержит чёткий предмет действия, нет размытых формулировок («улучшить», «повысить» без уточнений) |
| M - Measurable | Измеримость               | Есть числовой показатель, процент, единица измерения или иной способ верификации                        |
| A - Achievable | Достижимость              | Цель реалистична для данной должности и подразделения (с учётом исторических данных)                    |
| R - Relevant   | Релевантность             | Цель связана с функцией должности, ВНД и стратегическими приоритетами компании                          |
| T - Time-bound | Ограниченность во времени | Указан конкретный срок или период выполнения                                                            |

**3.1.2 Методология оценки целей**

Команды могут использовать SMART как базовую методологию оценки целей. При желании допускается применение альтернативных или дополнительных фреймворков: OKR (Objectives & Key Results), FAST (Frequently discussed, Ambitious, Specific, Transparent) или комбинированного подхода. Выбор методологии должен быть обоснован в презентации решения.

|------------------|--------------------|-----------------------------------------|
| **Выход**        | **Формат**         | **Описание**                            |

| Итоговый индекс  | Float 0.0--1.0     | Агрегированный SMART-score              |

**3.2.2 Логика алертов и предупреждений**

Процесс генерации включает следующие шаги:

1.  Retrieval (RAG): по должности и подразделению сотрудника система извлекает релевантные фрагменты ВНД и стратегических документов из векторной базы знаний.

2.  Контекстуализация: к найденным фрагментам добавляются данные о сотруднике (должность, уровень, подразделение), фокус-приоритеты квартала и, при наличии, цели вышестоящего руководителя.

3.  Генерация (LLM): языковая модель формирует 3--5 предложений целей, каждое из которых сразу проверяется на SMART-соответствие. Если SMART-индекс ниже порогового значения (например, 0.7), цель автоматически переформулируется.

4.  Верификация: итоговые цели проходят проверку на дублирование с уже существующими целями сотрудника.

**3.2.3 Логика алертов и предупреждений**

|                          |                  |                                                      |
|--------------------------|------------------|------------------------------------------------------|
| **Выход**                | **Формат**       | **Описание**                                         |

| Источник каждой цели     | Text + ссылка    | Название ВНД / раздел стратегии                      |

| Контекст генерации       | Text             | Краткое пояснение, почему предложена именно эта цель |

# 5. Архитектура решения {#архитектура-решения .unnumbered}

Решение строится на трёхуровневой архитектуре:

|                       |                                                |                                                                          |
|-----------------------|------------------------------------------------|--------------------------------------------------------------------------|
| **Уровень**           | **Компонент**                                  | **Функция**                                                              |
| Уровень данных        | Document Ingestion Pipeline                    | Загрузка, чанкование и векторизация ВНД; хранение целей и истории оценок |
| Уровень AI            | Goal Generator + Quality Evaluator (RAG + LLM) | Оценка целей по SMART, генерация предложений, переформулировка           |
| Уровень представления | HR Dashboard + Employee UI + Alert Manager     | Визуализация, интерфейс выбора целей, уведомления                        |

## 5.1 Поток данных {#поток-данных .unnumbered}

- Оценка: Цель сотрудника → Quality Evaluator → JSON с оценками и рекомендациями → UI-карточка

- Генерация: Профиль сотрудника → RAG-запрос к векторной базе ВНД → LLM-генерация → SMART-проверка → Список предложений → Выбор сотрудником

- Аналитика: Все оценки → Агрегация → Дашборд качества целей по подразделениям и кварталам

- Уведомления: Низкий индекс качества или отсутствие стратегической связки → уведомление руководителю и сотруднику

# 9. Заключение {#заключение .unnumbered}

Разработка AI-модуля оценки и генерации целей решает одну из ключевых болей HR-функции и руководителей: формальное, размытое, стратегически несвязанное целеполагание. Внедрение модуля создаёт измеримую ценность сразу на трёх уровнях: операционном (экономия времени), качественном (рост доли стратегически связанных целей) и стратегическом (выравнивание целей сотрудников со стратегией и ВНД компании).

Синтетические данные, предоставленные организаторами, позволяют участникам с первых часов хакатона приступить к разработке моделей. Прототип, созданный на хакатоне, может быть непосредственно адаптирован для промышленного внедрения в любую HR-систему через REST API.

## 10. Передача результатов и исходного кода {#передача-результатов-и-исходного-кода .unnumbered}

### Состав передаваемых материалов {#состав-передаваемых-материалов .unnumbered}

По завершении хакатона все команды обязуются передать организатору:

- исходный код реализованных сервисов и библиотек;

- обученные модели (если использовался AI/ML), файлы весов и скрипты обучения/инференса;

- вспомогательные скрипты для подготовки данных, запуска и тестирования решений;

- файлы конфигураций (docker-compose, .env.example, настройки БД и т. п.);

- краткую техническую документацию по сборке и запуску (README).

### Форма передачи {#форма-передачи .unnumbered}

Передача осуществляется в виде архива (zip/tar.gz) или доступа к репозиторию (GitHub/GitLab и т. п.). В репозитории или архиве должен быть минимальный, но достаточный набор инструкций, позволяющий собрать и запустить решение по шагам.

### Конфиденциальные данные и секреты {#конфиденциальные-данные-и-секреты .unnumbered}

В передаваемых материалах не должно быть логинов, паролей, ключей доступа к внешним сервисам и иной чувствительной информации. Все такие параметры должны быть вынесены в конфигурационные файлы или переменные окружения с примерами значений.

## 10.1 Права на использование разработок {#права-на-использование-разработок .unnumbered}

### [Авторские права]{.mark} {#авторские-права .unnumbered}

[Все авторские и исключительные права на исходный код, модели и иные результаты, созданные в рамках хакатона, остаются за участниками команды, если иное не будет отдельно согласовано с организатором в письменной форме.]{.mark}

### Лицензия для организатора {#лицензия-для-организатора .unnumbered}

Как условие участия в хакатоне, команда предоставляет организатору бессрочную, безотзывную, мировую, безвозмездную, неисключительную лицензию на использование созданного решения и его частей, включая:

- использование внутри компании организатора;

- модификацию, доработку и интеграцию в другие системы;

- создание производных решений на основе переданного кода и алгоритмов;

- развёртывание в тестовой и промышленной среде.

[При этом команда сохраняет право самостоятельно и независимо развивать и использовать своё решение, в том числе для других проектов и заказчиков.]{.mark}

### Цель использования {#цель-использования .unnumbered}

Организатор вправе применять полученные разработки и наработки для масштабирования, развития и внедрения в собственные бизнес-процессы и ИТ-системы, а также для последующей внутренней и внешней демонстрации результатов хакатона (без раскрытия коммерческих тайн участников).
